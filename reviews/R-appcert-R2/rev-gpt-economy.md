# R-appcert-R2 Economy Review — GPT-5.5

## Verdict

CONDITIONAL_APPROVE

R2 的经济设计方向总体成立：它没有把认证、MCP、WASM 执行和游戏经济混成一个隐式系统，而是明确区分了身份成本、运行成本、部署成本、资源物流和账号生命周期。这个方向像 Screeps 的成功经验加上 EVE/Factorio 式物流约束：让“自动化能力”进入游戏，但用可计量资源、时间、可见性和维护曲线限制无限扩张。

条件是：在进入实现前，需要把几类“看起来只是参数、实际会决定经济生死”的机制固化成更强的设计合同。尤其是 PoW 难度治理、全局存储/市场反垄断、实体膨胀惩罚、账号删除资产处置和证书生命周期成本，目前有框架但缺少足够硬的边界。

## Strengths

- 资源模型分层合理：本地存储、全局存储、转换时间、转换损耗和 Terminal/市场规则把“抽象经济力量”与“物理物流”区分开，避免全局钱包变成即时战略中的瞬移补给。
- 维护曲线方向正确：drone lifespan、active aging、Controller 维修总上限、Depot 资源维护成本共同形成持续消耗，能抑制静态囤兵和永久前线。
- 反雪球意识明确：新玩家低密度出生、避免包围、safe_mode、soft_launch、World 无排行榜、Arena 有公平窗口，这些都接近已验证的 MMO/RTS 反滚雪球组合。
- MCP 与 WASM 的经济边界清晰：MCP 不直接做 move/attack/build，AI 与人类都必须通过 WASM 进入世界，避免“API 玩家”天然拥有无成本高频操作优势。
- 证书生命周期设计较成熟：用途隔离、短 TTL、设备级吊销、CodeSigningCertificate 过期不影响已部署模块，减少了凭证更新对游戏经济连续性的破坏。
- 账号删除开始触及资产处置：abandon/recycle/transfer 三种策略比“删除账号即凭空蒸发/复制资产”更接近可运营的经济系统。

## Concerns

### A1 — High — PoW 经济性缺少自适应治理与攻击成本闭环

注册/CSR 依赖 PoW，文档说明 difficulty 可配置，但没有看到按服务器压力、注册失败率、IP/ASN/设备信誉、账号年龄或资源消耗自适应调整的规则。固定 PoW 在 AI agent 原生接入场景里很容易两头不讨好：对移动端和新人过贵，对批量云端 agent 太便宜。

建议把 PoW 从“一个 challenge 参数”升级为“准入成本策略”：

- 定义 difficulty_bits 的默认区间、最大值、移动端降级路径和服务器过载时的升降速率。
- 绑定 challenge 获取限流、CSR 提交限流、失败重试退避和同一 public key / username 前缀 / IP 段的批量成本。
- 给出目标经济指标，例如正常人类注册 P50/P95 耗时、单云核每小时可批量注册账号上限、攻击者每 1000 个账号的最低成本。
- 明确 PoW 只防批量注册，不替代后续游戏内资源成本、部署限流和行为惩罚。

### A2 — High — 全局存储税是好方向，但默认参数可能不足以阻止市场垄断

全局存储累进税、转换损耗和 No Teleport 都是正确机制，但默认税率很轻，且税基只看容量占比。富玩家可以通过多账号、多世界、联邦身份、本地仓库分散、挂单拆分或 Terminal 网络规避全局税。若市场交易可即时使用全局存储，少数早期玩家仍可能垄断流动性和关键资源价格。

建议补齐：

- 税基同时考虑“账户簇/受益人/交易关联图”，至少在审计层识别多账号拆仓。
- 市场挂单收取保证金、成交费、取消冷却或库存占用费，避免零成本刷单和流动性封锁。
- 对稀缺资源加入 velocity / hoarding 指标，而不只看余额占容量。
- 明确 Arena、Tutorial、World 三类世界的市场是否隔离，避免新手经济被持久世界资本输入污染。

### A3 — High — 实体膨胀惩罚可能形成“富玩家外部化成本”

快照上限、truncation、MAX_VISIBLE_ENTITIES 和 visibility_abuse 检测很有必要，但当前响应包含“降低 COMBAT 优先级”和“snapshot_quota -= 10%”。这类惩罚如果主要伤害被挤压的观察者，而不是制造实体膨胀的人，会变成富玩家向周围玩家外部化信息成本：用大量低价值实体污染对手视野，使对手 snapshot 截断、path_find 降级或战斗判断失真。

建议把实体膨胀成本内生化：

- 对“可见实体污染源”的 owner 收取 upkeep / command budget / spawn cap 成本，而不是主要削减受害者 snapshot。
- 区分正常大规模帝国和恶意低价值实体海：按实体价值、密度、重复行为、生命周期和战术贡献评分。
- 对 neutral/dropped resources/廉价 drone 的可见性截断优先级和 owner 成本建立明确规则。
- 将滥用响应从“COMBAT 优先级”改为更可解释的经济惩罚：维护费上升、spawn 冷却、局部实体上限或资源回收折损。

### A4 — Medium — 维护曲线有骨架，但缺少 late-game 反雪球函数

drone aging、Controller 维修上限和 Depot 维护能防止“永久单位”，但帝国规模扩大后的边际成本函数还不完整。RCL、房间数、跨房间物流、Terminal/市场、Observer 视野会给大玩家复合优势；如果没有 empire-level upkeep，老玩家优势会从线性增长变成网络效应增长。

建议定义随规模上升的成本曲线：

- 每额外 controlled room 的行政维护费、Controller downgrade 压力或物流损耗。
- 跨房间资源调拨的距离成本、拥堵成本和拦截风险。
- Observer/Terminal/Depot 网络的带宽或冷却上限，避免大帝国拥有近实时全图经济调度。
- 新玩家区域附近的高等级玩家投射能力衰减，而不仅是 safe_mode 时间窗。

### A5 — Medium — 账号删除资产处置容易被套利

删除账号支持 abandon/recycle/transfer，方向正确，但当前 transfer “接收方签名确认 + 全部资产转移”会变成资产迁移、规避税、规避战争损失或洗号套利的通道。recycle 默认 50% 退还到最近 Spawn 也可能被用作战时瞬时撤资或跨空间压缩物流。

建议把账号删除视为经济事件而不是纯认证事件：

- 删除进入 grace period 后资产冻结，不应继续参与市场、战斗、维护收益或可转移收益。
- transfer 应有冷却、审计、税费、资产估值快照和战争/封锁状态限制。
- recycle 应明确资源返还位置、延迟、生效 tick、是否可被拦截，以及与本地/全局存储转换规则一致。
- username 释放不得导致历史资产、声誉、市场记录或联邦映射混淆。

### A6 — Medium — 证书生命周期成本对 AI agent 群体偏低

证书 TTL、续签和多设备规则很完整，但从经济视角看，证书续签、设备扩张和 agent fleet 运维几乎没有游戏内/运营内成本。AI 玩家可长期维持大量账号、设备证书和部署能力；如果仅靠 PoW 和 deploy 限流，可能不足以约束“低成本策略迭代 + 多账号试探”。

建议增加非安全性的经济/运营约束：

- 每账号 active device / active certificate 上限，超出需要更高信任或运营审核。
- CodeSigningCertificate 续签与 deploy 频率绑定风控，而不是只验证持钥。
- 对高频 deploy / rollback / validate_module 施加资源费、冷却或编译队列优先级成本。
- 将 agent fleet 异常指标纳入准入策略：同源注册、相同模块 hash 批量部署、同构行为图等。

### A7 — Low — 接口读起来直观，但经济参数分散，容易实现时漂移

经济相关参数散在 gameplay、engine、tick、gateway、auth、mcp-tools 等文档中。新人能理解概念，但很难知道“最终权威默认值”在哪里，尤其是 deploy 限流在不同文档中同时出现 10/h、1/tick、10 tokens/s、50 MCP 请求/tick 等表达。

建议建立单一 Economy Parameters Registry：

- resource/storage/market/upkeep/deploy/auth/rate-limit 的默认值集中列出。
- 每个参数标注 world.toml key、适用模式、单位、是否可热更新、最小/最大值和安全含义。
- 文档中其它章节只引用 registry，避免多个默认值漂移。

## Missing

- 缺少完整的经济仿真基准：新手 0→RCL3、老玩家 10 房间帝国、前线战争 1000 tick、市场垄断、账号批量注册、实体膨胀攻击等场景的资源流入/流出曲线。
- 缺少参数边界：许多 world.toml 项可配置，但没有最小/最大值、禁用后果和“官方世界不得低于”的安全下限。
- 缺少市场微结构：订单簿、手续费、挂单保证金、价格操纵检测、成交结算位置、跨世界资产是否可交易尚未形成闭环。
- 缺少多账号/联邦套利模型：联邦身份隔离做得不错，但经济层还未说明如何识别或接受 sybil 资产拆分。
- 缺少资产冻结语义：删除、封禁、证书吊销、CVE 暂停部署、账号恢复期间，资产是否 aging、是否交税、是否可被攻击/占领，需要统一状态机。

## Phase Ordering

1. 先冻结 Economy Parameters Registry：把所有默认值、单位、上下限、模式差异和权威来源统一，否则后续实现会在多个文档间漂移。
2. 再补准入经济：PoW 自适应、注册/CSR/deploy/validate 限流、agent fleet 风控和证书数量上限要先定，因为它们决定 sybil 成本。
3. 然后补资源闭环：全局/本地转换、市场手续费、存储税、维护费、跨房间物流和 empire upkeep 必须能在仿真里守恒。
4. 接着补资产状态机：删除、恢复、封禁、吊销、transfer、recycle、abandon、grace period 的资产冻结与 tick 生效规则要统一。
5. 最后做平衡仿真与红队场景：用固定世界参数跑新手成长、老玩家扩张、市场垄断、实体膨胀、AI 批量注册和战争后勤，验证是否仍满足反雪球目标。

## Approval Conditions

- A1、A2、A3 至少需要设计级补丁后才能进入实现阶段。
- A4、A5、A6 可作为同轮设计补丁或下一轮 R3 前置项，但不能留到编码后再“调参数”。
- A7 是文档治理问题，不阻塞架构方向，但建议立刻处理以降低实现歧义。
