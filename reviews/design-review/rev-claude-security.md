# Security Review — Swarm DESIGN.md (Phase 0)

**Reviewer**: Claude Opus 4.8 (Security) · 信任边界 / 攻击面 / 威胁穷举
**Stage**: 2 — 终审
**评审视角**: 信任边界在哪里？每个玩家可控字段都是潜在武器。不是找 bug，是找"攻击者会怎么想"。

---

## VERDICT

**条件性放行（Conditional Approve）** — 设计可进入实现，但 C1/C2/C3 必须在编码动工前于 DESIGN.md 中给出收口方案，否则反作弊基石与核心进程完整性无法成立。

架构的信任分层方向是对的：WASM 不可信 / Rhai 可信 / Rust 核心，三层职责清晰，攻击面被有意识地切割。问题不在分层本身，而在**两条信任边界的执行细节**：

1. **确定性边界被墙钟终止击穿**（C1）——回放再验证是整个反作弊体系的根，一旦确定性不成立，后面所有审计、取证、申诉机制全部失去地基。这是结构性的，不是补丁能糊住的。
2. **可信层的"可信"假设过强**（C2）——"服主可信"在模组被分发、复用、二次打包后必然崩塌，而 Rhai 持破坏性 API 且在核心进程内裸跑，逃逸即全盘沦陷。
3. **唯一收口点的鲁棒性 = 整个攻击面的鲁棒性**（C3）——Command 校验管线是不可信 WASM 与可信世界状态之间的唯一闸门，它的穷举完备性决定一切。

三条 Critical 都集中在"信任从哪里进入系统"这一点上，恰好印证分层架构本身是健康的——缺陷是可定位、可收口的边界执行问题，而非需要推倒重来的根本性误设。因此给条件性放行而非打回。

---

## Strengths（值得肯定的设计决策）

- **信任分层显式且自洽**：WASM 不可信 / Rhai 可信 / Rust 核心三层，把"谁能改世界状态"这一最关键的权限收敛到了 Rust 核心，玩家代码（WASM）只能"提议"（构造 Command）不能"执行"。这是正确的能力最小化方向。
- **不可信玩家代码只输出意图、不直接改状态**：WASM 构造 Command JSON、由核心校验后落地的"提议—审批"模型，天然把攻击面压到单一 JSON 接口上。只要 C3 收口做扎实，攻击面是可枚举、可测试的。
- **确定性核心 + 回放的设计意图正确**（§1.5）：把反作弊建立在"可复现"之上，方向比事后启发式检测更扎实。问题只是 C1/M4 让这个意图当前无法兑现——目标对、实现待修。
- **fuel 计量已被纳入设计**（§1.3）：对玩家侧计算成本计费的意识是有的，H1 只是补齐"引擎侧代付成本"这一缺口，框架已在。
- **种子洗牌引入随机执行序**（D2）：用 Blake3 打乱玩家执行顺序以抑制固定序位优势，思路正确，H4 只需把 world_seed 保密性 / commit-reveal 补上。
- **MIT 首日开源 + 审计日志（TickTrace）**：透明度本身是安全资产，配合可复现回放可形成强取证链——前提是 I2 的防篡改与隐私分级落地。

## Critical

- **C1｜Rhai 墙钟终止破坏确定性**：§8.4 以 100ms 墙钟强制终止并回滚。墙钟在不同硬件/负载下终止于不同 AST 节点 → 同一输入产生不同世界状态。直接违反 §1.5「确定性核心」，使回放再验证（反作弊根基）失效。攻击者可借机器负载差异制造不可复现的状态分歧。**必须改为 AST 节点数 / 指令数等确定性预算**，墙钟只能作为兜底告警而非状态决定因素。
- **C2｜Rhai 在引擎进程内、无隔离、持破坏性 API**：§8.1/D11，`damage_entity` / `deduct_resource` / `set_entity_flag` 运行于核心进程。"服主可信"假设在模组被分发 / 复用时崩塌（D12 无版本约束、无签名）。Rhai 解释器一旦存在逃逸或预算绕过，即全进程沦陷。**需进程 / 能力隔离 + 模组签名与来源校验**。
- **C3｜Command Validation Pipeline 为单点收口**：§1.4/§1.3 Phase 2a。Command JSON 完全由不可信 WASM 构造。须穷举：entity_id 越权（操控非己方实体）、u32 数量溢出 / 回绕、资源数量注入、坐标越界、超大 / 畸形 JSON 致 DoS。**每条指令须对当前世界状态做所有权绑定 + 范围 + 类型三重校验，且校验自身不可被指令数量耗尽**。

## High

- **H1｜快照序列化未计入 fuel 配额**：§1.3 COLLECT 阶段引擎为每玩家序列化可见世界 JSON 并写入线性内存，成本由引擎承担、不受 fuel 计量。攻击者扩大可见范围 / 实体数 → 放大序列化与内存开销，形成非对称 DoS。需对快照大小 / 序列化成本设上限并计费。
- **H2｜MCP 查询工具无频率限制**：D10，`swarm_get_snapshot` / `swarm_inspect_entity` 无频控声明。高频轮询既是信息不对称（D9）也是 DoS 向量。须配额化所有查询工具。
- **H3｜Hack/Neutral 5-tick 状态机 TOCTOU**：D7。Hack 期间 drone 被 kill/Recycle、双重 Hack、中断恢复竞态 → 状态错乱或所有权丢失 / 复制。需形式化状态机并枚举并发边缘。
- **H4｜种子洗牌可预测**：D2，`Blake3(tick || world_seed)`。若 world_seed 已知 / 可推断，玩家预先计算自身执行序位并抢跑。须保证 world_seed 不可获取，或采用 commit-reveal。

## Medium

- **M1｜转换损耗整数舍入**：§5.2 transfer cost 1%/5% 对小额转账可舍入为 0 → 微额拆分规避损耗。需向上取整或设最小损耗。
- **M2｜回放 / 观战侧信道**：D14，`spectate_delay=0` 默认值构成共谋通道（一人观战全图、一人参赛）。竞技模式默认应 >0。
- **M3｜自定义动作组合注入**：§4.5 TOML `[[custom_actions]]` 引用 `[[special_effects]]`，缺乏组合白名单 → 服主误 / 恶配出免疫叠加或负冷却等破坏性组合。
- **M4｜wasmtime 版本即确定性依赖**：§8.2 自承「确定性依赖 wasmtime 版本」。版本漂移使旧回放无法复现 → 反作弊取证断裂。须锁定并记录运行时版本于回放元数据。

## Informational

- **I1｜MIT 首日开源**：须明确 wasmtime / Rhai CVE 跟踪与更新策略。
- **I2｜TickTrace 审计日志须防篡改**，并明确回放中的 PII / 隐私分级落地。
- **I3｜`set_entity_flag` 标志命名空间**（如 `immune_*`）需防冲突与未授权赋予。

---

## 放行条件（必须在实现前于 DESIGN.md 收口）

1. C1：墙钟 → 确定性指令预算，回放再验证路径明确。
2. C2：Rhai 隔离方案（进程 / 能力沙箱二选一）+ 模组签名与来源校验机制。
3. C3：Command 校验三重校验（所有权 + 范围 + 类型）的字段级穷举表，含 DoS 上限。

满足以上三条，安全方向给予正式放行。
