## Critical Findings
- [A-C1] Forbidden technology references remain in target documents — Files: design/architecture.md §8; design/engine.md §3; specs/core/tick-protocol.md §3.5.6; specs/core/persistence-contract.md §5.3; specs/core/snapshot-contract.md §7.2 — BLOCKER: `Dragonfly`, `ClickHouse`, `Rhai`, and `fdb` references appear in architecture and core contracts. The review rules classify any Rhai/Dragonfly/ClickHouse/FoundationDB reference as BLOCKER, and the references also preserve replacement/history language inside target-state documents. Remove the forbidden names entirely, including test names, variable names, capacity formulas, and replacement tables.
- [A-C2] Sharding authority model is mutually inconsistent — Files: design/architecture.md §2; design/architecture.md §6; design/engine.md §3.1a "扩展策略声明"; specs/core/shard-protocol.md §2; specs/core/shard-protocol.md §5 — Architecture and engine design define static coordinate-range sharding with `shard_id = f(room_x, room_y)`, one Engine process per shard, and one `.redb` file per shard. The shard protocol instead uses `room_id` with Jump Hash and later states that all shards commit inside one Engine process to the same `.redb` file. These cannot all be the authoritative routing, storage, and failure-domain model. Pick one sharding contract and update all files to use the same shard key, assignment algorithm, process boundary, redb boundary, and replay identity.
- [A-C3] Cross-shard hot-path semantics contradict the architecture boundary — Files: design/architecture.md §2; design/architecture.md §10; design/engine.md §3.1a "扩展策略声明"; specs/core/shard-protocol.md §3; specs/core/shard-protocol.md §4 — The architecture says cross-shard communication occurs only for player migration and is not hot-path. The shard protocol adds cross-shard visibility and `RangedAttack`, including a two-phase combat protocol with one-tick eventual consistency. This changes tick lifecycle, combat determinism, visibility snapshots, and latency guarantees. Define whether cross-shard combat/visibility exists in the target architecture; if it does, move the hot-path protocol into architecture and tick lifecycle rather than leaving it as a conflicting shard-only rule.
- [A-C4] Deploy activation depends on incompatible storage and sandbox availability rules — Files: specs/core/persistence-contract.md §2.3; specs/core/tick-protocol.md §2.4; specs/core/tick-protocol.md §9.2; specs/core/distributed-sandbox.md §5; specs/core/distributed-sandbox.md §7 — Persistence says activation is allowed when the compiled artifact matches, and pending object-store upload does not block activation. Tick protocol §9.2 says `upload_status != "complete"` makes the deploy fail at N+1. Distributed sandbox then allows `ModuleNotFound` on cache miss after a successful deploy. These rules produce different observable tick results and replay inputs. Make redb deploy manifest plus compiled artifact the single activation authority, and specify a deterministic prefetch/fallback rule that cannot turn a successfully activated module into an arbitrary zero-command tick.
- [A-C5] Replay-critical snapshot/keyframe storage is undefined across persistence contracts — Files: design/architecture.md §6a; design/engine.md §3.2; design/engine.md §3.4.7; specs/core/persistence-contract.md §2.1; specs/core/persistence-contract.md §2.2; specs/core/persistence-contract.md §10.2; specs/core/incremental-snapshot.md §4; specs/core/incremental-snapshot.md §5 — The persistence contract says deterministic replay needs TickCommitRecord plus keyframe/delta chain while object storage is non-replay-critical; design/engine says keyframes are stored to redb; incremental snapshot says keyframes are in the blob store and also under `snapshots/keyframe/{world_id}/{tick}`; persistence §10.2 says keyframes are independent files under `$REDB_PATH.keyframes`. This leaves replay dependent on three different storage models. Define one authoritative keyframe/delta storage layer and update replay-critical/non-critical classifications accordingly.

## High Findings  
- [A-H1] TickCommitRecord schema and field count are inconsistent — Files: specs/core/persistence-contract.md §2.1; specs/core/persistence-contract.md §7.1; specs/core/tick-protocol.md §3.5.5; specs/core/tick-protocol.md §3.5.6; specs/core/tick-protocol.md §9.4; design/engine.md §3.3 — Persistence §2.1 declares exactly 10 replay-critical TickCommitRecord fields, §7.1 adds `collect_id`, `attempt_id`, and `commit_id`, tick protocol introduces `GlobalTickCommit` and TickInputEnvelope boundaries, and engine design lists additional replay identity and environment fields. Implementers cannot tell which fields are mandatory in redb, which are envelope/debug fields, or which hash chain they belong to. Publish one canonical schema with explicit replay-critical, envelope, retry identity, and rich/debug partitions.
- [A-H2] Fuel refund and crash/retry semantics conflict — Files: specs/core/tick-protocol.md §3.3; specs/core/tick-protocol.md §3.5.6 "COLLECT 结果跨重试缓存"; specs/core/tick-protocol.md §6.1; specs/core/tick-protocol.md §8.4; specs/core/tick-protocol.md §9.4 — Fuel exhaustion and WASM crashes do not refund, redb retry reuses cached COLLECT fuel and refunds after abandon, Phase 2a panic keeps fuel, redb commit failure refunds, and crash recovery before commit can fully refund or retry from cached state depending on section. This affects player billing, replay, and deterministic recovery. Define one state machine for COLLECT completed, EXECUTE started, commit retry, tick abandon, and process crash, including whether cached COLLECT fuel is final, reused, or refunded.
- [A-H3] Perception snapshot timing is contradictory — Files: specs/core/snapshot-contract.md §1.1; specs/core/tick-protocol.md §2.3; specs/core/tick-protocol.md §9.3; design/engine.md §3.2; design/engine.md §3.4.4 — Snapshot contract says the per-player perception snapshot is generated at each tick end. Tick protocol and engine design say it is built at COLLECT start and shared by WASM and MCP before EXECUTE. This changes what WASM can observe and what snapshot hash means in replay. Make COLLECT-start generation the single timing rule or rewrite tick lifecycle/replay around end-of-tick snapshots.
- [A-H4] Command ordering has competing canonical definitions — Files: specs/core/tick-protocol.md §3.1; specs/core/tick-protocol.md §9.1; specs/security/command-source.md §8.1; design/engine.md §3.2 — Tick protocol §3.1 uses seeded player shuffle plus per-player sequence; §9.1 uses `(priority_class, shuffle_index, source_rank, sequence, command_hash)` and includes `MCP_Deploy`; command-source §8.1 says deploy is not in RawCommand and orders Admin, WASM, then MCP_Query by source/submission order. These can produce different replay order for Admin/TestHarness/Tutorial/MCP classes and ties. Designate one canonical sorting key for gameplay commands and separately define deploy/control-plane replay ordering.
- [A-H5] NATS cluster wording conflicts with the no-cluster premise — Files: design/architecture.md §2; design/architecture.md §3; design/architecture.md §7; design/architecture.md §10 — The sharding model says there is no cluster, leader election, or gossip, but the architecture diagram and message-bus section define NATS as a cluster and extension strategy includes NATS cluster nodes. If "no cluster" only means "no Engine consensus cluster", the text should say so. Otherwise the deployment model contradicts itself.
